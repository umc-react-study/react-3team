## 이 프로젝트에서 비교하는 차이점

### 1) Error Boundary vs 일반 try/catch

| 항목      | Error Boundary                               | try/catch                         |
| --------- | -------------------------------------------- | --------------------------------- |
| 잡는 범위 | 렌더링 중 발생한 React 트리 에러             | 함수/이벤트 핸들러 내부 로직 에러 |
| 사용 위치 | 컴포넌트 트리 상단                           | 비동기 함수, 버튼 클릭 핸들러 등  |
| 주 용도   | 앱 전체 크래시 방지, 대체 화면 제공          | API 호출/저장 로직 실패 처리      |
| 예시 파일 | `src/components/common/AppErrorBoundary.tsx` | `src/App.tsx`의 `handleSave`      |

핵심: **UI 렌더링 안정성은 Error Boundary**, **동작 로직 실패는 try/catch**가 담당합니다.

---

### 2) 선언적 에러 처리 vs 명령적 에러 처리

| 항목 | 선언적 처리                                       | 명령적 처리                         |
| ---- | ------------------------------------------------- | ----------------------------------- |
| 방식 | 상태(`isLoading`, `error`, `data`)에 따라 UI 분기 | `try/catch`에서 직접 성공/실패 분기 |
| 장점 | 화면 상태 흐름이 읽기 쉬움                        | 특정 액션 흐름 제어가 명확함        |
| 단점 | 상태 관리 코드가 늘어날 수 있음                   | 로직이 이벤트 내부에 집중됨         |
| 예시 | `DeclarativeUserSection`                          | `ImperativeActionSection`           |

핵심: **화면 중심이면 선언적**, **행동 중심(저장/삭제/전송)이면 명령적**이 보통 더 자연스럽습니다.

---

### 3) API 에러 분류: 예측 가능 vs 예측 불가능

| 항목      | 예측 가능한 에러             | 예측 불가능한 에러                  |
| --------- | ---------------------------- | ----------------------------------- |
| 예시      | 400/401/403/404/409/429      | 네트워크 단절, 서버 예외, 파싱 실패 |
| 처리 전략 | 에러 코드별 메시지/UX 분기   | 공통 안내 + 재시도/문의 유도        |
| 코드 구조 | `mapHttpError`에서 코드 매핑 | `UNKNOWN`으로 통합 처리             |
| 예시 파일 | `src/lib/errors.ts`          | `src/lib/http.ts`                   |

핵심: **예측 가능한 에러는 세분화**, **예측 불가능한 에러는 안전한 기본 처리**로 가져갑니다.

---

### 4) Fallback UI 전략

`ErrorFallback` 컴포넌트에서 에러 타입별로 다른 UI를 보여줍니다.

- `UNAUTHORIZED`: 로그인 필요 안내
- `FORBIDDEN`: 권한 없음 안내
- `NOT_FOUND`: 데이터 없음 안내
- 기타/알 수 없음: 공통 오류 메시지 + 재시도 버튼

예시 파일: `src/components/common/ErrorFallback.tsx`

## 실무에서 주로 언제 쓰는가

### Error Boundary

- 페이지 전체가 깨질 수 있는 주요 화면(라우트, 대시보드, 상세 페이지) 상단에 둡니다.
- 외부 라이브러리 UI(차트, 에디터, 복잡한 위젯)로 인해 렌더링 에러 가능성이 있을 때 사용합니다.
- 목적은 "에러를 없애는 것"이 아니라, **크래시를 격리하고 사용자 이탈을 줄이는 것**입니다.

### try/catch (명령적 처리)

- 버튼 클릭, 저장/삭제, 폼 제출처럼 **사용자 액션 기반 비동기 작업**에 사용합니다.
- 성공/실패 후 행동이 달라야 할 때(토스트, 모달 닫기, 재시도 등) 적합합니다.
- 서버 호출, 파일 업로드, 결제 요청 같은 I/O 작업에서 기본 패턴으로 씁니다.

### 선언적 에러 처리

- 목록/상세 조회처럼 **화면 상태 자체가 중요한 페이지**에서 유리합니다.
- `loading/error/empty/success` 상태를 분기해서 UI를 일관되게 유지할 때 사용합니다.
- React Query, SWR 같은 데이터 패칭 라이브러리와 조합할 때 특히 효과적입니다.

### API 에러 분류 (예측 가능/불가능)

- 인증/권한/검증 에러처럼 자주 발생하고 대응이 정해진 경우는 **예측 가능한 에러**로 분류합니다.
- 네트워크 단절/서버 장애처럼 원인이 불명확한 경우는 **예측 불가능 에러**로 공통 처리합니다.
- 이 분류를 먼저 해두면 화면마다 중복 분기하지 않고 재사용 가능한 UX를 만들기 쉽습니다.

### Fallback UI

- "무조건 동일 문구" 대신, 사용자 행동이 가능한 메시지(재시도, 로그인 유도, 문의 안내)를 제공합니다.
- 기능별 실패 지점마다 다른 폴백 수준을 둡니다.
  - 컴포넌트 단위 실패: 해당 카드/섹션만 대체
  - 페이지 단위 실패: 페이지 전체 폴백
  - 앱 단위 실패: Error Boundary 전역 폴백

### 빠른 선택 기준

- 렌더링 중 터짐 방지: **Error Boundary**
- 클릭/저장/전송 실패 처리: **try/catch**
- 조회 화면 상태 표현: **선언적 처리**
- 사용자 안내 품질 향상: **API 에러 분류 + Fallback UI**

## 파일 구조

```txt
src/
  components/common/
    AppErrorBoundary.tsx   # 렌더링 에러 보호
    ErrorFallback.tsx      # 에러 코드별 폴백 UI
  lib/
    errors.ts              # 에러 타입/분류 규칙
    http.ts                # fetch 래퍼 + 분류 연결
  App.tsx                  # 선언적/명령적 처리 예시 화면
  main.tsx                 # 루트에 Error Boundary 적용
```

## 테스트 시나리오

1. **Error Boundary 테스트**
   - `Error Boundary 테스트(강제 크래시)` 버튼 클릭
   - 앱이 완전히 죽지 않고 대체 UI가 노출되는지 확인
2. **선언적 처리 테스트**
   - `유저 조회` 버튼 클릭
   - 로딩 -> 성공 데이터(or 실패 fallback) 흐름 확인
3. **명령적 처리 테스트**
   - `저장 시뮬레이션(409 응답)` 버튼 클릭
   - `CONFLICT` 메시지로 실패 안내되는지 확인
